# Лог изменений и решений — guillotine_optimizer

## Задача
Оптимизатор раскладки деталей на лист методом гильотинных разрезов.
Лист: 743×580 мм. Деталей: 59 шт. (11 типов).

---

## Сессия 1 — Анализ и сборка рабочего варианта

### Проблема: mod-декларации ломали сборку
Файл `src/main.rs` содержал строки:
```rust
mod main_ok_2; mod main1; mod main_FINAL_CLEAN;
```
Rust пытался подключить резервные файлы как модули, что вызывало дублирование структур и функций.

**Решение:** Удалены три строки `mod ...` из начала `main.rs`.

---

### Добавлен Python-скрипт визуализации (`visualize.py`)
**Что делает:**
- Читает `result_task_N.json`
- Рисует PNG-схему: цветные прямоугольники по типам деталей
- Подписывает имя, размер (WxH), площадь внутри каждой детали
- Размерные стрелки листа, легенда, заголовок с эффективностью
- Автоматически запускается из Rust после завершения задачи

**Решение при битом JSON:** Добавлена проверка в `load_json()` — при ошибке парсинга выводится содержимое файла для диагностики.

## Сессия 2 — Многоагентная оптимизация алгоритма

### Запуск workflow из трёх агентов (параллельно в git worktrees)
Цель: улучшить количество размещённых деталей.

**Агент A** — улучшение `try_fill_gaps` и добавление `try_local_swap`:
- `try_fill_gaps`: убран шаг `step = w.min(h).max(5)`, теперь перебор с шагом 1 (исчерпывающий поиск)
- Добавлена функция `try_local_swap`: убрать 1 размещённую деталь, попробовать разместить 2+ меньших

**Агент B** — расширение beam width на глубоких уровнях + бонус за делимость:
- Beam на глубинах 12+: 5→8, 4→6, 3→4
- В `score_move` добавлен бонус за делимость размеров

**Результат агента B: ХУЖЕ** — эффективность упала с 98.11% до 97.92%.
**Причина:** Более широкий beam на глубоких уровнях = медленнее = меньше охват за то же время.

**Решение:** Откат изменений агента B. Оставлены только изменения агента A.

### Итоговые beam widths (оставлены оригинальные):
```rust
depth < 3:   beam_width (полный)
depth < 7:   beam_width * 3/4, min 8
depth < 12:  5
depth < 18:  4
иначе:       3
```

## Сессия 3 — Принудительное первое размещение (forced first placement)

### Проблема: 72×42 не размещается
Лучший результат — 58/59 деталей (98.11%). Не размещается деталь `72×42` (3024 мм²).

**Анализ:**
- Суммарная площадь всех 59 деталей: 425,822 мм²
- Площадь листа: 430,940 мм²
- Теоретический потенциал: 59/59 возможен (остаток 5,118 мм² = 1.19%)
- Но: после раскладки 58 деталей нет ни одной гильотинно-доступной позиции для 72×42

### Решение: блок принудительного поиска (добавлен в `src/main.rs`)
После основного поиска и `try_local_swap`, если есть неразмещённые детали:
1. Для каждой неразмещённой детали × 2 ориентации × 2 типа разреза = N вариантов
2. Деталь ставится в позицию (0,0), создавая два подпрямоугольника
3. `solve_recursive` заполняет каждый подпрямоугольник из полного набора деталей
4. Все варианты запускаются параллельно через rayon
5. Timeout: 40 сек, beam width: удвоенный (min 24)

**Параметры разрезов для 72×42:**
- H-разрез: R1=671×42 + R2=743×538. НЕВОЗМОЖЕН: 58 деталей (419,774 мм²) > R2 (399,734 мм²)
- V-разрез: R1=72×538 + R2=671×580. ВОЗМОЖЕН: суммарно 427,916 мм² > 419,774 мм²

**Результат:** Улучшений не найдено. Лучший результат остался 98.11% (58/59).

### Вывод
Гильотинный алгоритм с beam search за 100 сек (60+40) не может найти раскладку 59/59.
98.11% — практический потолок для данной конфигурации при текущих ограничениях времени.

Для попытки достичь 59/59 рекомендуется:
```bash
./target/release/guillotine_optimizer -f config.json -t 300 -b 20
```
